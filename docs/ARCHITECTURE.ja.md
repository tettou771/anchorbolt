## 1つのバイナリ、2つの役割

AnchorBolt は、3つのサブコマンドを持つ単一の実行ファイルとして提供されます：

- **`anchorbolt start`** — 会場側のスーパーバイザー。運用環境（`TRUSSC_MCP`、`TRUSSC_MCP_PORT`、`TRUSSC_LOG_FILE`）を注入した状態でアプリを起動し、監視し、再起動し、そして `--server` を指定すればハートビート、サムネイル、ログを手元へ送ります。
- **`anchorbolt serve`** — フリートサーバー。ダッシュボード、取り込みエンドポイント、DB を使わないストレージ、WebSocket コマンドハブ、そして AI 向けの `/mcp`。
- **`anchorbolt token`** — サーバー側で2種類のトークンを発行・失効させます。

ワイヤープロトコルの両端が1つのコードベースに含まれているため、会場とそのサーバーの間でバージョンがずれることは構造的に起こりえません。

すべての設定の優先順位：**フラグ ＞ `ANCHORBOLT_TOKEN` 環境変数 ＞ 設定ファイル ＞ デフォルト。** 設定はコメントを許容する JSON で、`anchorbolt start` は `anchorbolt.json` の隣にあればそれを自動で読み込みます。

## プロセス監視（`start`）

スーパーバイザーはアプリを**2つの独立した方法**で監視します：

1. **プロセスの終了** — `waitpid`（POSIX）／プロセスハンドル（Windows）。アプリが死んだので再起動します。
2. **ハング** — 標準の `tc_get_health` MCP ツールへの正常な応答が `--watchdog-timeout` 秒（実時間、デフォルト10秒）返ってこない。プロセスは生きているが固まっているので、終了させて再起動します。

ウォッチドッグは、応答回数のミスカウントではなく、最後に正常応答があった時点からの実時間で測ります。だから HTTP のタイムアウトによって実効的な監視ウィンドウが伸びることはありません。起動猶予期間（`--grace`、デフォルト15秒）は、起動の遅いアプリが最初の正常報告を返す前に殺されないよう守ります。`--watchdog-timeout 0` はプロセスの終了だけを監視します——これこそが、MCP エンドポイントを持つ TrussC アプリだけでなく、**あらゆる**バイナリに対して AnchorBolt を役立たせている仕組みです。

終了は、まず穏やかに、それから強制的に：POSIX では SIGTERM → 5秒 → SIGKILL、Windows ではアプリのウィンドウへの WM_CLOSE → 5秒 → TerminateProcess。再起動は短いバックオフを待ち、`restart` イベントは検知時点ではなく*実際に再起動した時点*で発火します——だから、OS のシャットダウンがスーパーバイザーの直前にアプリを殺した場合でも、記録されるのは静かな `stop` だけで、幻のインシデントにはなりません。

**ポート衝突への安全策：** `tc_get_health` はアプリの pid を返し、スーパーバイザーはそれを自分の子プロセスと照合します。もし別のアプリが共有ポートで応答した場合、それは死んだ子プロセスを覆い隠すのではなく「ミス」として数えられます。MCP ポート自体もデフォルト値から上へスキャンされるので、複数のスーパーバイザーが1台のマシンで共存できます。

## ログ配送 — オフラインスプール

会場のログこそが信頼できる情報源であり、常に完全で、日次ファイルとしてローカルに保持されます。`--server` が設定されている場合は送信もされます——そしてその送信は、数時間平気で途切れる会場ネットワークの現実に耐えるよう作られています。

ローカルの日次ファイルが、そのままスプールです。永続化されたカーソル（`push-cursor.json`）が、サーバーがどこまで受信を*確認*したかを記録し、200 が返ったときだけ前進します。だから、ネットワーク断も、AnchorBolt の再起動も、マシンの再起動さえも、何一つ失いません——再接続時には、カーソルから未送分がまとめて送られます。セマンティクスは**at-least-once（少なくとも1回）**です：送信中のバッチは凍結され、バイト単位でそのまま再送され、サーバーは `(ファイル, バイトオフセット)` で重複を除去します。だから、ACK の消失が生むのは高々1つの重複であって、欠落は決して生みません。まだ受信確認されていないファイルは、その古さに関わらずローカルの間引き処理から保護されます。

カーソルは遅延書き込みされます（約60秒ごと＋終了時、一時ファイルへ書いてから rename するアトミック方式）。だから、絶えず更新されるポインタが SD カードを摩耗させることはありません。クラッシュで最後の書き込みを失っても、再送されるのは高々1分ぶんの既配送行だけで、それはサーバーが破棄します。

決定的に重要なのは、**ログの送信がアプリのヘルス状態から独立している**ことです。アプリがハングしている間も、スーパーバイザー自身の「無反応／再起動中」という行はサーバーへ流れ続けます——だから、ハングしているアプリ（＝対処中）は、暗転してしまったマシンと、遠隔からでも区別できます。

## フリートサーバー（`serve`）

`serve` は、HTTP サーバー（同梱の httplib による）と小さな WebSocket ハブをホストするヘッドレスな TrussC アプリです。以下に応答します：

- **取り込み**（エージェントから）：`POST /api/heartbeat`（ヘルス JSON）、`POST /api/thumb/<id>` と `POST /api/image/<id>/<name>`（生の JPEG）、`POST /api/log/<id>` と `POST /api/alert/<id>`（JSON 行）。
- **読み取り**（ダッシュボード用）：`GET /api/apps`、`/api/thumb/<id>`、`/api/image/<id>/<name>`、`/api/history/<id>`（ハートビートの時系列）、`/api/log/<id>?after=<seq>`、`/api/alert/<id>?after=<seq>`。
- **コマンドブリッジ**：`POST /api/command/<id>` はリクエストを WebSocket ハブ経由で会場へ中継し、応答が返るまでブロックします。
- **ダッシュボード**（`/`）——外部アセットのない、自己完結した1枚の HTML ページ。

ダッシュボードは**ウォール**（会場ごとのライブサムネイル、緑／赤、タブでグループ化）を表示し、クリックすると**詳細ビュー**を表示します：より大きなライブサムネイル、fps／メモリ／カスタムのグラフ、カスタム画像ストリーム、イベント一覧、そして重要度で色分けされ検索可能なログパネル。

**インシデントバッジ：** 会場のあらゆるイベント（`restart` / `up` / `down` / `update` / `stop` / `alert`）が詳細ビューのイベント一覧に載り、インシデント（`restart` / `down` / `alert`）はさらに、オペレーターが Clear をクリックするまでウォールのカードに赤いバッジを点灯させます——だから「osaka が夜のうちに3回再起動した」ことが、翌朝に何も開かなくても目に入ります。

## ストレージと保持期間

ストレージは**設計上 DB を使わず**、`--data`（デフォルト `./anchorbolt-data`）以下に置かれます：

- アプリごとの日次ハートビート／ログ／アラート JSONL（`heartbeat-YYYY-MM-DD.jsonl` など）
- タイムスタンプ名の JPEG（`thumbs/YYYYMMDD-HHMMSS.jpg`、および名前ごとのカスタム画像）——ディレクトリ*そのもの*がインデックスです
- `tokens.json`、`operators.json`、`groups.json`、`sessions.json`、`codes.json`

インスタレーション規模（1アプリあたり1日数 MB）なら、「昨夜の2時ごろのエラー」という問いに、ファイルスキャンが ripgrep の速さで答えます。追記専用ファイルはクラッシュに強く、マイグレーション不要で、他のすべてが壊れているときでも `less` で読めます。

**保持期間**（`--keep-days`、デフォルト90、0 = 無期限）は、区切りより古い JSONL と画像を1時間ごとに間引きます。サムネイルにはさらに追加の間引きがあります：直近24時間は完全に保持し、それより古いショットは1時間あたり1枚まで薄めます。これは**会場ローカルの `--log-keep` とは独立**しています——削除がどちらの方向にも伝播することはなく、サーバーは通常キオスクよりも*長く*データを保持します（事後調査こそがサーバーの存在理由だからです）。すべての間引きはファイル名の文字列計算で行われ、mtime もパースも使いません。

## コマンドチャンネル（WebSocket）

インタラクティブな機能——Restart／Update ボタン、ライブビュー、リモートコントロール、AI パススルー——では、サーバーが会場から受け取るだけでなく、会場に*話しかける*必要があります。これは WebSocket 上で動きます：

- 各会場はサーバーのハブへ**1本の外向き WebSocket** を張り続けます（NAT に優しく、会場が外へダイヤルするので会場側に受信ポートは不要）。
- ハブは TrussC の `TcpServer` の上に作られた最小限の RFC6455 サーバーで、**別のポート**（サーバーポート＋1、デフォルト 54723）で動きます——同梱の httplib は WebSocket をサポートしないため、ハブは独立したものになっています。
- サーバー→会場は `{type:"cmd", id, action, …}`、応答は `{type:"result", id, ok, …}` です。`POST /api/command/<id>` は HTTP リクエストをこれに橋渡しし、対応する応答が返るまでブロックします（タイムアウト15秒）。

単一のホスト名を公開するリバースプロキシやトンネルの背後では、会場は `host:port+1` に到達できません。`https://` のサーバー URL は「前段に TLS 終端プロキシが居る」ことを示すので、会場は規約に従いハブを **`wss://<host>/ws`** と導出します——そのパスをプロキシでハブへルーティングしてください（ハブはハンドシェイクの際にパスを無視するので、サーバー側の変更は不要）。変則的なプロキシのパスだけ **`--ws-url`** を明示します。いずれの場合も監視は HTTP のルートだけで完全に機能します——ハブを使うのはインタラクティブな機能だけです。

## リモートアップデート

`--allow-update` を指定すると、ダッシュボードの **Update** ボタンが会場マシン上でパイプラインを実行します——デフォルトでは `git pull --ff-only` → `trusscli update` → `trusscli build` で、設定の `update` 配列で上書きできます（先頭に `trusscli upgrade` を足せば TrussC 自体も更新でき、TrussC 以外のプロジェクトには任意のコマンドを使えます）。

パイプラインは**古いバイナリを動かしたまま**実行されるので、ビルドの失敗がインスタレーションを止めることは決してありません——ビルド出力はダッシュボードのログパネルへライブでストリームされ、アプリは成功したときにだけ新しいバイナリで再起動します。前のバイナリは `<binary>.prev` として保持されます：**Roll back** ボタンでそれを復元でき、また新しいバイナリが初回起動で正常にならなければ自動でロールバックされます。

2つの気の利いた仕様があります：`git pull` が何も新しく持ってこなければ、デフォルトのパイプラインはそこで止まりアプリには触れません（「念のため」に Update を押すのはタダです）。そして失敗した試行はどのコミットで失敗したかを覚えているので、同じコミットでのリトライはスキップされず最後まで実行されます。エージェントはハートビートのたびにプロジェクトの git コミットを報告するので、ウォールにはどの会場がどのバージョンを動かしているかが表示されます。

これは定義上リモートコード実行なので、会場側での明示的な `--allow-update` オプトインが必要です。

## ライブビューとリモートコントロール

**Live** ボタンはアプリの画面をストリームします：エージェントが `tc_get_screenshot` をポーリングし、JPEG フレームを同じコマンド WebSocket 上へ（約10fps で）押し出します。これはブラウザが見ている間だけ行われます——フレームエンドポイントへのポーリングがストリームを開始し、ブラウザが離れると10秒後にエージェントが自動で停止します。明示的なライフサイクル管理は一切ありません。

**control** をオンにすると、ブラウザのポインタ／キーボードイベントがアプリへマッピングされます：クリックとドラッグは `tc_mouse_*` に、キー入力は `tc_key_press`（sokol キーコード）になり、ホバーはチャンネルを溢れさせないよう約20回／秒にスロットルして転送されます。だから、ホバーに反応するインスタレーションもきちんと反応します。座標は、縮小されたフレームではなくアプリの実際のウィンドウサイズ（ヘルスから取得）でスケールされます。

コントロールには、operator ロール（サーバー側）と `--allow-control`（会場側）の**両方**が必要で、さらにアプリが `mcp::registerDebuggerTools()` で入力注入にオプトインしていなければなりません。

## 認証

トークンは2種類、どちらもサーバーで発行され、一度だけ表示され、SHA-256 ハッシュとしてのみ保存されます。

**エージェントトークン**（`tokens.json`、`{app-id: hash}`）——会場の、送信専用のアイデンティティ。**オープンモードはありません**：すべての取り込みリクエストと WebSocket の hello が認証を通らなければならず、fleet はデフォルトで閉じていて、URL を知っているだけでは何もできません。id は発行時にトークンに束縛され、会場は id を自分で名乗るのではなくトークンから導出します（`POST /api/whoami`、ハッシュの逆引き）——だから `--id` は存在せず、エージェントトークンが漏れたときの被害範囲もその会場1つの偽データに限られ、別の id へのなりすましはできません。`--server` を指定しない会場はこれをすべてスキップします（ローカル専用の監視にトークンは不要）。

**オペレータートークン**（`operators.json`、`{name: {role, hash, created, scope}}`）——ダッシュボードにいる人間（と AI）。ロールは **viewer**（読み取り専用）、**operator**（＋再起動／アップデート／コントロール／アラートのクリア）、**admin**（＋設定ページ）。このクラスには*オープンモードがあります*：オペレーターが1人も登録されていなければダッシュボードは開放状態です（ブートストラップ経路——最初の admin を発行するために設定ページへ到達できるように）。オペレーターが1人でも登録されていれば、ダッシュボードはログインを要求します。トークンは **HttpOnly クッキー**に載るので（だから `<img>` のサムネイルも認証を通せます。これは `Authorization` ヘッダーではできないことです）、リクエストのたびにハッシュと再照合されます——だから、あるオペレーターを失効させれば、次のポーリング時にロックアウトされ、無効化すべきセッションテーブルもありません。

**スコープ**はオペレーターが見られる範囲を狭めます：グループ名または `app:<id>` エントリの配列で、空はすべてを意味します。スコープ外のアプリはどこでも 404 になります——ウォール、アプリごとの全エンドポイント、そしてフリートの `/mcp`——だから、会場の存在そのものすら、スコープを絞られたクライアントには漏れません。

**セッションと6桁コード。** コード（`codes.json`、使い切り、TTL 10分、10回失敗すると 429 を返すグローバルなブルートフォース対策付き）は2つのフローに使われます：

- **ペアリング** — `anchorbolt start --pair <code>` は、そのコードを（認証なしで。コード*そのもの*が認証です）新たに発行されたエージェントトークンと引き換え、会場マシン上に非公開で保存します（プラットフォームの状態ディレクトリ内の `anchorbolt.token.json`、パーミッション 0600、バイナリ名をキーとするので、以降の素の実行ではマシンがそれを再利用します）。`tc-...` の文字列が手作業でコピーされることは一切ありません。
- **ログインコード** — オペレーター用に1つ発行すると、6桁を入力するだけでサインインできます。これは `os-...` セッショントークン（`sessions.json`）を発行し、それが `operators.json` を通して解決されるので、オペレーターを失効させればそのセッションも自動で無効になります。

## フリートMCP（AIオペレーション）

serve ポート上の `POST /mcp` は完全な MCP サーバーです（TrussC アプリと同じ、HTTP 上の JSON-RPC トランスポート）。だから、AI アシスタントをフリートに向けることができます：

```bash
claude mcp add --transport http anchorbolt https://ops.example.com/mcp \
    --header "Authorization: Bearer op-..."
```

ツール：`list_apps`、`search_logs`、`tail_logs`、`get_events`、`get_health_history`、`list_screenshots`、`get_screenshot`（本物の MCP イメージブロックを返すので、アシスタントはインスタレーションを*見る*ことができます）、`restart_app`、そして各アプリ自身の MCP ツールへの `app_list_tools` / `app_call` パススルー。パススルーは、すべてのアプリのツールをフリートのツール一覧にミラーするのではなく、会場を引数に取る2つの固定プロキシを使います——だから、何台の会場が接続しても一覧は固定のままです。ロールは適用されます：viewer には読み取り専用のツールが与えられ、`restart_app` や変更を伴う `app_call` には operator が必要です（さらに会場の `--allow-control` がその上に重ねて効きます）。

> **オープンモードの注意：** オペレーターが1人も登録されていないと、`/mcp` は全員に admin を与えます——つまり、URL を知っている誰もがリモート再起動できてしまいます。サーバーを公に公開する*前に*、オペレーターの admin トークンを登録してください。

## アプリが公開するステータスとアラート

TrussC アプリは、値1つにつき1行で、自前の監視データを公開できます——スーパーバイザーの設定は不要で、スーパーバイザーが MCP の `tools/list` を通じてツールを発見します：

```cpp
void tcApp::setup() {
    mcp::status("scene",         [&] { return sceneName; });     // shown as-is
    mcp::statusGraph("visitors", [&] { return visitorCount; });  // plotted over time
    mcp::statusImage("entranceCam", [&] { return camPixels; });  // e.g. a webcam
    mcp::alert("IR camera disconnected!");                        // → sinks + dashboard
}
```

`status` の値はハートビートのたびに載り、`statusImage` のストリームはサムネイルの間隔で取得され、`alert` は標準の `tc_get_alerts` ツールから汲み出されて、通知 sinks とダッシュボードのイベント一覧へ配られます。これは意図的に notify ではなく **alert** と名付けられています——人間が知っておくべき出来事に対して上げるものであって、メッセージバスとして使うものではありません。

## プラットフォーム別メモ

- **macOS / Linux** — 完全にサポートされています。ログはデフォルトで `~/Library/Logs/anchorbolt/<id>/`（macOS）または `$XDG_STATE_HOME/anchorbolt/<id>/`（Linux）に出力されます。このプラットフォーム慣例の場所は、launchd/systemd 下で作業ディレクトリが `/` になっても問題ありません。
- **Windows** — スーパーバイザーは `KILL_ON_JOB_CLOSE` を持つジョブオブジェクト内で CreateProcess を使うので、アプリのプロセスツリー全体がスーパーバイザーとともに終了します（AnchorBolt が強制終了されても、フルスクリーンウィンドウが孤児として残りません）。ログは `%LOCALAPPDATA%\anchorbolt\<id>\` に出力されます。リモートアップデートは、実行中の exe が書き込みロックされる一方でリネームは可能である、という事実を考慮しています（バックアップはリネームで行い、リンカーのためにパスを解放します）。CI ではコンパイルが通っており、実機での検証は進行中です。
- 外向きの JSON はすべて、不正な UTF-8 を例外ではなく U+FFFD に置き換えてダンプされます——ローカライズされたツールチェーンが CP932 のビルド出力をログへ流し込んでも、スーパーバイザーをクラッシュさせてはならないからです。

## 設計上の決定

**`trusscli` と分離する。** 開発 CLI の仕事はプロジェクトのビルドであり、見守って手元へ報告するような振る舞いからは切り離しておきます——npm と pm2 の分業と同じです。おせっかいはビルダーにとって鬱陶しく、運用エージェントにとってはそれこそが価値のすべてで、両者の対象はほとんど重なりません（ワークショップの初心者が AnchorBolt に出会うことはなく、インスタレーションのプロは2つ目のツールを厭いません）。

**データベースではなく、ファイル。** 履歴は必須要件なので、ストアは常に存在します。追記専用の JSONL ＋タイムスタンプ付きの JPEG は、クラッシュに強く、マイグレーション不要で、grep でき、デーモンも要りません。データベース（FTS インデックスとしての SQLite）は、長期間にわたる検索に対する*後々の*最適化として考えうるものであって、ファイルを信頼できる情報源に据えたままにします——初日からの依存物ではありません。

**OAuth ではなく、トークン。** オフライン動作は必須要件です——不安定なネットワーク上の会場も動き続けなければならず、サーバーは Google/SSO への依存なしにプロジェクトごとにデプロイできなければなりません。ハッシュ化されたトークンは、即時の失効（ここではステートレス性より重要です）と、オフラインでのハッシュ照合をもたらします。SSO がどうしても必要になった場合は、AnchorBolt の中ではなくリバースプロキシで終端します。

**サービスごとのアダプターではなく、1つのテンプレート化された Webhook エンジン。** Slack、Discord、ntfy、Kuma、そして任意の汎用エンドポイントは、いずれもボディテンプレートを持つ同じ HTTP POST であり、プリセットはそれを事前に埋めるだけです。サービスを追加するとは、アダプターを追加することではなく、プリセットを追加することです。

**通知は一方向で、対話はダッシュボードに置く。** Webhook は外へ押し出しますが（集約・重複除去済み）、変更を伴うアクションの承認、アプリの操作、ログの閲覧——それらはダッシュボードで起こることで、チャットアプリの中では決して起こりません。その道は、サービスごとのボットと OAuth の泥沼へと続いています。

**ローカルファースト。** ほとんどの会場が必要とするのは自動再起動とローカルログだけで、サーバーはフリートを運用する少数のためのものです。すべてのサーバー機能はオプションです——素の `anchorbolt start` は、それ自体で完結した、役立つツールです。
