## 1つのバイナリ、2つの役割

AnchorBolt は、4つのサブコマンドを持つ単一の実行ファイルとして提供されます：

- **`anchorbolt start`** — 会場側のスーパーバイザー。運用環境（`TRUSSC_MCP`、`TRUSSC_MCP_PORT`、`TRUSSC_LOG_FILE`）を注入した状態でアプリを起動し、監視し、再起動し、そして `--server` を指定すればハートビート、サムネイル、ログを手元へ送ります。
- **`anchorbolt serve`** — フリートサーバー。ダッシュボード、取り込みエンドポイント、DB を使わないストレージ、WebSocket コマンドハブ、そして AI 向けの `/mcp`。
- **`anchorbolt token`** — サーバー側で2種類のトークンを発行・失効させます。
- **`anchorbolt approvals`** — サーバーマシン上で AI の承認キューを一覧・裁定します。

ワイヤープロトコルの両端が1つのコードベースに含まれているため、アプリとそのサーバーの間でバージョンがずれることは構造的に起こりえません。

すべての設定の優先順位：**フラグ ＞ `ANCHORBOLT_TOKEN` 環境変数 ＞ 設定ファイル ＞ デフォルト。** 設定はコメントを許容する JSON で、`anchorbolt start` は `anchorbolt.json` の隣にあればそれを自動で読み込みます。

## プロセス監視（`start`）

スーパーバイザーはアプリを**2つの独立した方法**で監視します：

1. **プロセスの終了** — `waitpid`（POSIX）／プロセスハンドル（Windows）。アプリが死んだので再起動します。
2. **ハング** — 標準の `tc_get_health` MCP ツールへの正常な応答が `--watchdog-timeout` 秒（実時間、デフォルト10秒）返ってこない。プロセスは生きているが固まっているので、終了させて再起動します。

ウォッチドッグは、応答回数のミスカウントではなく、最後に正常応答があった時点からの実時間で測ります。だから HTTP のタイムアウトによって実効的な監視ウィンドウが伸びることはありません。起動猶予期間（`--grace`、デフォルト120秒）は、起動の遅いアプリが最初の正常報告を返す前に殺されないよう守ります。`--watchdog-timeout 0` はプロセスの終了だけを監視します——これこそが、MCP エンドポイントを持つ TrussC アプリだけでなく、**あらゆる**バイナリに対して AnchorBolt を役立たせている仕組みです。

終了は、まず穏やかに、それから強制的に：POSIX では SIGTERM → 5秒 → SIGKILL、Windows ではアプリのウィンドウへの WM_CLOSE → 5秒 → TerminateProcess。再起動は短いバックオフを待ち、`restart` イベントは検知時点ではなく*実際に再起動した時点*で発火します——だから、OS のシャットダウンがスーパーバイザーの直前にアプリを殺した場合でも、記録されるのは静かな `stop` だけで、幻のインシデントにはなりません。

**ポート衝突への安全策：** `tc_get_health` はアプリの pid を返し、スーパーバイザーはそれを自分の子プロセスと照合します。もし別のアプリが共有ポートで応答した場合、それは死んだ子プロセスを覆い隠すのではなく「ミス」として数えられます。MCP ポート自体もデフォルト値から上へスキャンされるので、複数のスーパーバイザーが1台のマシンで共存できます。

## ログ配送 — オフラインスプール

アプリのログこそが信頼できる情報源であり、常に完全で、日次ファイルとしてローカルに保持されます。`--server` が設定されている場合は送信もされます——そしてその送信は、数時間平気で途切れる会場ネットワークの現実に耐えるよう作られています。

ローカルの日次ファイルが、そのままスプールです。永続化されたカーソル（`push-cursor.json`）が、サーバーがどこまで受信を*確認*したかを記録し、200 が返ったときだけ前進します。だから、ネットワーク断も、AnchorBolt の再起動も、マシンの再起動さえも、何一つ失いません——再接続時には、カーソルから未送分がまとめて送られます。セマンティクスは**at-least-once（少なくとも1回）**です：送信中のバッチは凍結され、バイト単位でそのまま再送され、サーバーは `(ファイル, バイトオフセット)` で重複を除去します。だから、ACK の消失が生むのは高々1つの重複であって、欠落は決して生みません。まだ受信確認されていないファイルは、その古さに関わらずローカルの間引き処理から保護されます。

カーソルは遅延書き込みされます（約60秒ごと＋終了時、一時ファイルへ書いてから rename するアトミック方式）。だから、絶えず更新されるポインタが SD カードを摩耗させることはありません。クラッシュで最後の書き込みを失っても、再送されるのは高々1分ぶんの既配送行だけで、それはサーバーが破棄します。

決定的に重要なのは、**ログの送信がアプリのヘルス状態から独立している**ことです。アプリがハングしている間も、スーパーバイザー自身の「無反応／再起動中」という行はサーバーへ流れ続けます——だから、ハングしているアプリ（＝対処中）は、暗転してしまったマシンと、遠隔からでも区別できます。

## フリートサーバー（`serve`）

`serve` は、HTTP サーバー（同梱の httplib による）と小さな WebSocket ハブをホストするヘッドレスな TrussC アプリです。以下に応答します：

- **取り込み**（エージェントから）：`POST /api/heartbeat`（ヘルス JSON）、`POST /api/thumb/<id>` と `POST /api/image/<id>/<name>`（生の JPEG）、`POST /api/log/<id>` と `POST /api/alert/<id>`（JSON 行）。
- **読み取り**（ダッシュボード用）：`GET /api/apps`、`/api/thumb/<id>`、`/api/image/<id>/<name>`、`/api/history/<id>`（ハートビートの時系列）、`/api/log/<id>?after=<seq>`、`/api/alert/<id>?after=<seq>`。
- **コマンドブリッジ**：`POST /api/command/<id>` はリクエストを WebSocket ハブ経由でアプリへ中継し、応答が返るまでブロックします。
- **ダッシュボード**（`/`）——外部アセットのない、自己完結した1枚の HTML ページ。

ダッシュボードは**ウォール**（アプリごとのライブサムネイル、緑／赤、タブでグループ化）を表示し、クリックすると**詳細ビュー**を表示します：より大きなライブサムネイル、fps／メモリ／カスタムのグラフ、カスタム画像ストリーム、イベント一覧、そして重要度で色分けされ検索可能なログパネル。

**インシデントバッジ：** アプリのあらゆるイベント（`restart` / `up` / `down` / `update` / `stop` / `alert`）が詳細ビューのイベント一覧に載り、インシデント（`restart` / `down` / `alert`）はさらに、オペレーターが Clear をクリックするまでウォールのカードに赤いバッジを点灯させます——だから「osaka が夜のうちに3回再起動した」ことが、翌朝に何も開かなくても目に入ります。

## ストレージと保持期間

ストレージは**設計上 DB を使わず**、`--data`（デフォルト `./anchorbolt-data`）以下に置かれます。分け方の基準はただ1つの問い——*これが消えたら、人間がやり直す羽目になるか？*——です：

```
anchorbolt-data/
├── config/                  # 人間が設定したもの — クリーンアップは手を付けない
│   ├── apps.json            #   {app-id: {token: <sha256>, group, hidden}} — 唯一の台帳
│   ├── operators.json       #   オペレーターアカウント（role、hash、scope）
│   ├── shares.json          #   共有リンク
│   └── notify.json          #   フリート全体の通知（Notify タブ）
├── state/                   # 機械が作るもの — `rm -rf state/` しても安全
│   ├── sessions.json  codes.json  approvals.json  apps-health.json
│   └── approval-decisions/
└── apps/<id>/               # アプリごとのコンテンツ — クライアント単位で削除できる
    ├── heartbeat-*.jsonl  log-*.jsonl  alert-*.jsonl
    ├── thumbs/              # タイムスタンプ名の JPEG — ディレクトリ*そのもの*がインデックス
    └── images/<name>/       # カスタム statusImage ストリーム
```

- **`config/`** は人間が設定したものすべて——アプリのトークン／グループ、オペレーター、共有リンク、通知設定。クリーンアップは決して触れません。サーバーを引っ越すなら `config/` をコピーすれば済みます。
- **`state/`** は自己再生します——セッション、ログインコード、承認キュー、ヘルスキャッシュ。削除しても、全員がログアウトされる以上のことは起きません。
- **`apps/<id>/`** がクライアントごとの大きなコンテンツです。案件が終わったら、そのディレクトリを1つ消すだけです。

旧リリースのデータディレクトリ（トップレベルにフラットにファイルが並ぶ形式）は、新しい `serve` の初回起動時に自動でこのレイアウトへ移行されます——手作業は不要です。

インスタレーション規模（1アプリあたり1日数 MB）なら、「昨夜の2時ごろのエラー」という問いに、ファイルスキャンが ripgrep の速さで答えます。追記専用ファイルはクラッシュに強く、マイグレーション不要で、他のすべてが壊れているときでも `less` で読めます。

**保持期間**（`--keep-days`、デフォルト90、0 = 無期限）は、区切りより古い JSONL と画像を1時間ごとに間引きます。サムネイルにはさらに追加の間引きがあります：直近24時間は完全に保持し、それより古いショットは1時間あたり1枚まで薄めます。これは**会場側の `--log-keep` とは独立**しています——削除がどちらの方向にも伝播することはなく、サーバーは通常キオスクよりも*長く*データを保持します（事後調査こそがサーバーの存在理由だからです）。すべての間引きはファイル名の文字列計算で行われ、mtime もパースも使いません。

## コマンドチャンネル（WebSocket）

インタラクティブな機能——Restart／Update ボタン、ライブビュー、リモートコントロール、AI パススルー——では、サーバーがアプリから受け取るだけでなく、アプリに*話しかける*必要があります。これは WebSocket 上で動きます：

- 各アプリのエージェントはサーバーのハブへ**1本の外向き WebSocket** を張り続けます（NAT に優しく、会場側から外へダイヤルするので会場側に受信ポートは不要）。
- ハブは TrussC の `TcpServer` の上に作られた最小限の RFC6455 サーバーで、**別のポート**（サーバーポート＋1、デフォルト 54723）で動きます——同梱の httplib は WebSocket をサポートしないため、ハブは独立したものになっています。
- サーバー→エージェントは `{type:"cmd", id, action, …}`、応答は `{type:"result", id, ok, …}` です。`POST /api/command/<id>` は HTTP リクエストをこれに橋渡しし、対応する応答が返るまでブロックします（タイムアウト15秒）。

単一のホスト名を公開するリバースプロキシやトンネルの背後では、会場側は `host:port+1` に到達できません。`https://` のサーバー URL は「前段に TLS 終端プロキシが居る」ことを示すので、エージェントは規約に従いハブを **`wss://<host>/ws`** と導出します——そのパスをプロキシでハブへルーティングしてください（ハブはハンドシェイクの際にパスを無視するので、サーバー側の変更は不要）。変則的なプロキシのパスだけ **`--ws-url`** を明示します。いずれの場合も監視は HTTP のルートだけで完全に機能します——ハブを使うのはインタラクティブな機能だけです。

## リモートアップデート

ダッシュボードの **Update** ボタン（operator、デフォルトで許可）は、会場マシン上でパイプラインを実行します——デフォルトでは `git pull --ff-only` → `trusscli update` → `trusscli build` で、設定の `update` 配列で上書きできます（先頭に `trusscli upgrade` を足せば TrussC 自体も更新でき、TrussC 以外のプロジェクトには任意のコマンドを使えます）。

パイプラインは**古いバイナリを動かしたまま**実行されるので、ビルドの失敗がインスタレーションを止めることは決してありません——ビルド出力はダッシュボードのログパネルへライブでストリームされ、アプリは成功したときにだけ新しいバイナリで再起動します。前のバイナリは `<binary>.prev` として保持されます：**Roll back** ボタンでそれを復元でき、また新しいバイナリが初回起動で正常にならなければ自動でロールバックされます。

2つの気の利いた仕様があります：`git pull` が何も新しく持ってこなければ、デフォルトのパイプラインはそこで止まりアプリには触れません（「念のため」に Update を押すのはタダです）。そして失敗した試行はどのコミットで失敗したかを覚えているので、同じコミットでのリトライはスキップされず最後まで実行されます。エージェントはハートビートのたびにプロジェクトの git コミットを報告するので、ウォールにはどのアプリがどのバージョンを動かしているかが表示されます。

これは定義上リモートコード実行なので、サーバー側の operator ロールでゲートされます。絶対に遠隔更新されたくないアプリは `--deny-update` でオプトアウトします。

## ライブビューとリモートコントロール

**Live** ボタンはアプリの画面をストリームします：エージェントが `tc_get_screenshot` をポーリングし、JPEG フレームを同じコマンド WebSocket 上へ（約10fps で）押し出します。これはブラウザが見ている間だけ行われます——フレームエンドポイントへのポーリングがストリームを開始し、ブラウザが離れると10秒後にエージェントが自動で停止します。明示的なライフサイクル管理は一切ありません。

**control** をオンにすると、ブラウザのポインタ／キーボードイベントがアプリへマッピングされます：クリックとドラッグは `tc_mouse_*` に、キー入力は `tc_key_press`（sokol キーコード）になり、ホバーはチャンネルを溢れさせないよう約20回／秒にスロットルして転送されます。だから、ホバーに反応するインスタレーションもきちんと反応します。座標は、縮小されたフレームではなくアプリの実際のウィンドウサイズ（ヘルスから取得）でスケールされます。

コントロールに必要なのは、operator ロール（サーバー側）と、アプリが `mcp::registerControlTools()` で入力注入にオプトインしていること——これがゲートのすべてで、会場側フラグは不要です。変更を伴うツールはアプリが登録したときにだけ存在するので、アプリ自身の MCP サーフェスが control のオプトインそのものです。エージェントはこれを申告する（`caps.control`）ので、ダッシュボードは該当するときだけ control トグルを出します。

## 認証

トークンは2種類、どちらもサーバーで発行され、一度だけ表示され、SHA-256 ハッシュとしてのみ保存されます。

**エージェントトークン**（`config/apps.json` 内、`{app-id: {token: <sha256>, group, hidden}}`——アプリの唯一の台帳で、各アプリのグループと表示状態も一緒に持ちます）——アプリの、送信専用のアイデンティティ。**オープンモードはありません**：すべての取り込みリクエストと WebSocket の hello が認証を通らなければならず、fleet はデフォルトで閉じていて、URL を知っているだけでは何もできません。id は発行時にトークンに束縛され、アプリは id を自分で名乗るのではなくトークンから導出します（`POST /api/whoami`、ハッシュの逆引き）——だから `--id` は存在せず、エージェントトークンが漏れたときの被害範囲もそのアプリ1つの偽データに限られ、別の id へのなりすましはできません。`--server` を指定しないアプリはこれをすべてスキップします（ローカル専用の監視にトークンは不要）。

**オペレータートークン**（`config/operators.json`、`{name: {role, hash, created, scope}}`）——ダッシュボードにいる人間（と AI）。ロールは **viewer**（読み取り専用）、**operator**（＋再起動／アップデート／コントロール／アラートのクリア）、**admin**（＋設定ページ）。このクラスには*オープンモードがあります*：オペレーターが1人も登録されていなければダッシュボードは開放状態です（ブートストラップ経路——最初の admin を発行するために設定ページへ到達できるように）。オペレーターが1人でも登録されていれば、ダッシュボードはログインを要求します。トークンは **HttpOnly クッキー**に載るので（だから `<img>` のサムネイルも認証を通せます。これは `Authorization` ヘッダーではできないことです）、リクエストのたびにハッシュと再照合されます——だから、あるオペレーターを失効させれば、次のポーリング時にロックアウトされ、無効化すべきセッションテーブルもありません。

**スコープ**はオペレーターが見られる範囲を狭めます：グループ名または `app:<id>` エントリの配列で、空はすべてを意味します。スコープ外のアプリはどこでも 404 になります——ウォール、アプリごとの全エンドポイント、そしてフリートの `/mcp`——だから、アプリの存在そのものすら、スコープを絞られたクライアントには漏れません。

**セッションと6桁コード。** コード（`state/codes.json`、使い切り、TTL 10分、10回失敗すると 429 を返すグローバルなブルートフォース対策付き）は2つのフローに使われます：

- **ペアリング** — `anchorbolt start --pair <code>` は、そのコードを（認証なしで。コード*そのもの*が認証です）新たに発行されたエージェントトークンと引き換え、会場マシン上に非公開で保存します（プラットフォームの状態ディレクトリ内の `anchorbolt.token.json`、パーミッション 0600、バイナリ名をキーとするので、以降の素の実行ではマシンがそれを再利用します）。`tc-...` の文字列が手作業でコピーされることは一切ありません。
- **ログインコード** — オペレーター用に1つ発行すると、6桁を入力するだけでサインインできます。これは `os-...` セッショントークン（`state/sessions.json`）を発行し、それが `config/operators.json` を通して解決されるので、オペレーターを失効させればそのセッションも自動で無効になります。

## フリートMCP（AIオペレーション）

serve ポート上の `POST /mcp` は完全な MCP サーバーです（TrussC アプリと同じ、HTTP 上の JSON-RPC トランスポート）。だから、AI アシスタントをフリートに向けることができます：

```bash
claude mcp add --transport http anchorbolt https://ops.example.com/mcp \
    --header "Authorization: Bearer op-..."
```

ツール：`list_apps`、`search_logs`、`tail_logs`、`get_events`、`get_health_history`、`list_screenshots`、`get_screenshot`（本物の MCP イメージブロックを返すので、アシスタントはインスタレーションを*見る*ことができます）、`restart_app`、そして各アプリ自身の MCP ツールへの `app_list_tools` / `app_call` パススルー。パススルーは、すべてのアプリのツールをフリートのツール一覧にミラーするのではなく、アプリを引数に取る2つの固定プロキシを使います——だから、いくつのアプリが接続しても一覧は固定のままです。ロールは適用されます：viewer には読み取り専用のツールが与えられ、`restart_app` や変更を伴う `app_call` には operator が必要です（変更を伴うツールはアプリが登録したときにだけ存在します）。

### 承認キュー

変更を伴う `/mcp` 呼び出し（`restart_app` と、`tc_get_*` 以外の `app_call`）は直接実行されません：**人間の承認**を待つキューに入ります。呼び出しは約20秒だけ決定を待ち——誰かがダッシュボードにいるとき、承認は数秒で済むことが多いので——未決なら pending チケット（`ap-xxxxxx`）を返します。AI は読み取り専用の `get_approval` ツールでそれをポーリングします。承認 → アクションは serve の内部で実行され、結果がチケットに載る。拒否 → `denied by <名前>`。`--approval-ttl`（デフォルト 900 秒）を過ぎても未決なら失効。読み取りツールは決してキューされません。

決定の方法は2つ、どちらも人間専用です——`/mcp` 側には意図的に approve ツールが存在しないので、AI が自分のリクエストを自己承認することは構造的にできません：

- **ダッシュボード：** スコープがそのアプリを含む operator 以上に、ヘッダーへ `approvals N` カプセルが現れます。パネルで Approve / Deny。
- **サーバーマシン：** `anchorbolt approvals list | approve [id] | deny [id]`。ID は一意なプレフィックスで指定できます（`approve 3f`）。pending が1件だけなら ID は省略可能です。CLI の決定は `state/approval-decisions/` 以下に1決定1ファイルで書かれ、serve のスイーパー（約2秒周期）が拾います——だから CLI が serve 自身の `state/approvals.json` 書き込みと競合することはなく、実行（エージェントの WebSocket が必要）は常に serve の内部で行われます。

キューは `state/approvals.json` に永続化され、serve の再起動を生き延びます。`--data` なしでデータディレクトリを見つけるために、CLI は serve が起動時に書く**ランタイムポインター**——プラットフォームの状態ディレクトリ（Linux は `~/.local/state/anchorbolt/`、macOS は `~/Library/Logs/anchorbolt/`、Windows は `%LOCALAPPDATA%\anchorbolt\`）内の `serve.json`（`dataDir`・`port`・`pid`）——を読みます。解決順は：明示的な `--data` > `./anchorbolt-data` > ポインター（そのディレクトリが実在する間だけ信用されます。serve の起動ごとに書き直され、削除はされないので、serve が停止していてもキューのファイルに対して CLI は動き続けます）。ポインターはユーザーごとです：CLI は serve プロセスと同じユーザーで実行してください。

`serve --auto-approve` で従来の直接実行に戻せます（20秒の待ちが割に合わない、信頼できる LAN など）。

> **オープンモードの注意：** オペレーターが1人も登録されていないと、`/mcp` は全員に admin を与えます——つまり、URL を知っている誰もがリモート再起動できてしまいます。サーバーを公に公開する*前に*、オペレーターの admin トークンを登録してください。承認キューはオープンモードでも機能しますが、ダッシュボードにいる誰もが承認できてしまいます。

## アプリが公開するステータスとアラート

TrussC アプリは、値1つにつき1行で、自前の監視データを公開できます——スーパーバイザーの設定は不要で、スーパーバイザーが MCP の `tools/list` を通じてツールを発見します：

```cpp
void tcApp::setup() {
    mcp::status("scene",         [&] { return sceneName; });     // shown as-is
    mcp::statusGraph("visitors", [&] { return visitorCount; });  // plotted over time
    mcp::statusImage("entranceCam", [&] { return camPixels; });  // e.g. a webcam
    mcp::alert("IR camera disconnected!");                        // → notifications + dashboard
}
```

`status` の値はハートビートのたびに載り、`statusImage` のストリームはサムネイルの間隔で取得され、`alert` は標準の `tc_get_alerts` ツールから汲み出されて、通知先とダッシュボードのイベント一覧へ配られます。これは意図的に notify ではなく **alert** と名付けられています——人間が知っておくべき出来事に対して上げるものであって、メッセージバスとして使うものではありません。

## 通知

外向きの通知はすべて、1つのテンプレート式 Webhook エンジンが配送します：通知先の各エントリ（内部名は「sink」）は URL と、`{{app}}` `{{event}}` `{{msg}}` `{{time}}` のプレースホルダを持つ任意の body テンプレートのことで、プリセット（slack / discord / ntfy / uptime-kuma）はそれを埋めるだけです。配送は通知先ごとのキューとバックオフを備えた at-least-once（少なくとも1回）です——ネットワーク断の間はイベントを保持して回復を待ち、4xx はそのイベントを破棄します（壊れた Webhook URL を永遠にリトライしてはいけないからです）。

このエンジンは**2つの場所**で動き、両者は設計上独立しています（両方に設定すれば両方に届きます）：

- **会場側** — アプリの `anchorbolt.json` にある `notify` 配列（従来の `sinks` キーも引き続き読めます）。サーバーが無くても機能します。これがローカルファーストの経路です。
- **サーバー側** — `config/notify.json`。ダッシュボードの **Notify** タブ（admin）で編集します。フリート全体を1箇所で設定できることが要点です：各エントリは `scope`（グループ名 / `app:<id>`、オペレーターの scope と同じ意味論、空 = 全アプリ）を持てるので、「クライアント A の Slack にはクライアント A のアプリだけ」がフィールド1つで済み、`app:<id>` 1つだけに絞ったエントリは、そのアプリのマシンに同じ通知を設定したのと同じ振る舞いになります。

サーバー側のエントリには、アプリが送ってくるすべてのイベント（`restart` / `up` / `down` / `update` / `stop` / `alert`——取り込み時点でファンアウトされます）に加えて、**サーバーだけが知りうる3つのイベント**が届きます：

- **`approval`** — 変更を伴う AI の呼び出しが承認キューに入った。
- **`offline` / `online`** — `--offline-after`（デフォルト60秒）を超えるハートビートの沈黙と、その回復。死んだマシンは自分の死を報告できません——沈黙が見えるのはサーバーだけです。ウォッチャーは初回スキャンでベースラインを取るだけなので、serve の再起動が昔に消えたアプリを再通知することはありません。

### 実際に何が、いつ通知されるのか

障害の種類ごとに信号を意図的に分けてあるので、Webhook が鳴る＝「見に行くべき」だけを意味します：

| 何が起きたか | 誰が気づけるか | 信号 |
|---|---|---|
| アプリのクラッシュ／ハング | スーパーバイザー（自分は生きている） | `restart` / `down` イベント＋⚠バッジ——数秒以内 |
| OS / UPS のシャットダウン（計画的） | スーパーバイザーが、終了する間際に | 灰色の静かな `stop` イベント。**その後の `offline` は通知されない** |
| 電源断・カーネルパニック・ネットワーク消失 | マシン上の誰にも不可能 | `--offline-after` の沈黙後に `offline`、文面は `(no stop received — unexpected)` |

微妙なのは真ん中の行です。正常なシャットダウン（systemd/launchd が SIGTERM を送る——UPS 起点のシャットダウンも同じ経路を通るので、アプリ側のコードは不要）は、沈黙が始まる前に `stop` イベントを発火します。サーバーはそれを覚えていて、後で offline のしきい値を越えたとき、*clean stop によって予告された沈黙*はウォールのドットを赤にするだけで **Webhook は送りません**（再起動時の `online` も同様に静かです）。stop の記録がない沈黙こそが怖い種類で、そちらは通知されます。つまり、毎晩電源を落とすインスタレーションは Slack を騒がせず、ブレーカーが落ちたときは呼び出されます。

トレードオフも正直に：マシンが正常にシャットダウンした後、*戻ってこなかった*場合、Webhook は鳴りません——ウォールの赤ドットだけが手がかりです。「10時までに必ず起動していること」を固く保証したいアプリには、uptime-kuma のモニター（scope `app:<id>`）を向けてください——Kuma はこのロジックとは独立に、push の欠如そのものに自分のスケジュールで警報を鳴らします。

サーバー側の uptime-kuma は、scope をちょうど1つの `app:<id>` に絞ることを要求します——そのアプリのハートビートが新鮮な間だけ push し、会場側の kuma エントリと同じ意味になります。より広い scope の kuma には健全な「healthy」の定義が無いため、読み込み時に警告して弾かれます。

Notify タブの各行には **Test** ボタンがあり、*編集中の内容のまま*同期的に発火して結果（HTTP ステータスまたは接続エラー）を報告します。タブからリンクされる `/help/notify` には、サービスごとの短い Webhook 取得手順があります。保存すると通知エンジンは即座に再構成されます——再起動は不要です。

## タイムアウトとしきい値

タイミング定数は1本の連鎖になっていて、それを正直に保つルールは：**各レイヤーは、その下のレイヤーより辛抱強くあること**。これにより、あるレベルの一時的なつまずきが、その上のレベルで「障害」に化けることを防ぎます。

| レイヤー | デフォルト | 理由 |
|---|---|---|
| ヘルスポーリング（スーパーバイザー → アプリの MCP） | `--watchdog-timeout` / 3 ≈ 3秒 | ウォッチドッグの窓の中に約3回のチャンスが入る頻度。コストは無視できる程度 |
| ハングウォッチドッグ（再起動） | 実時間10秒 | 最後の正常応答からの実時間——HTTP タイムアウトが窓を引き伸ばすことはない |
| 起動猶予 | 120秒 | 重いアプリ（シェーダーのウォームアップ、モデル読み込み）は最初の正常応答までに15秒を平気で超える |
| 送信の HTTP タイムアウト（会場 → サーバー） | 10秒 | Raspberry Pi クラスのマシン＋トンネル越しの実測から：2秒だと偽の「unreachable」が連発した——*遅い*応答は*死んだ*サーバーではない。送信は監視サイクルに載っているので、1回の遅い往復は上のレイヤーより短く収まる必要がある |
| `offline` ステータス（赤ドット＝通知イベント） | `--offline-after` 60秒 | しきい値は**1つ**：ウォールはサーバーの offline 判定——通知を発火させるのと同じフラグ——をそのまま表示するので、「赤」と「通知が飛んだ」は単一の状態です。死んだマシンは自分で報告できない、持続する沈黙だけがそれを示す。ハートビート約20回分なのでトンネルの瞬断では誰も呼び出さず、本当のクラッシュは down/restart の通知でもっと早く届く。clean `stop` に予告された沈黙（計画的シャットダウン）はドットを赤にするだけで通知しない——通知の節を参照 |
| 承認の待ち時間 | 約20秒、以降はチケット（TTL 900秒） | 承認の多くは誰かがダッシュボードにいる間に済む。それを過ぎたら AI は接続を握り続けず `get_approval` でポーリングする |

チューニングするときは **ポーリング < 送信タイムアウト < offline** の順序を守ってください。そして、ウォールのドットは表示に過ぎないことを忘れずに——監視（再起動）と通知（down/alert）はそれぞれ独自の時計を持ち、ダッシュボードの見た目とは無関係に発火します。

## プラットフォーム別メモ

- **macOS / Linux** — 完全にサポートされています。ログはデフォルトで `~/Library/Logs/anchorbolt/<id>/`（macOS）または `$XDG_STATE_HOME/anchorbolt/<id>/`（Linux）に出力されます。このプラットフォーム慣例の場所は、launchd/systemd 下で作業ディレクトリが `/` になっても問題ありません。
- **Windows** — スーパーバイザーは `KILL_ON_JOB_CLOSE` を持つジョブオブジェクト内で CreateProcess を使うので、アプリのプロセスツリー全体がスーパーバイザーとともに終了します（AnchorBolt が強制終了されても、フルスクリーンウィンドウが孤児として残りません）。ログは `%LOCALAPPDATA%\anchorbolt\<id>\` に出力されます。リモートアップデートは、実行中の exe が書き込みロックされる一方でリネームは可能である、という事実を考慮しています（バックアップはリネームで行い、リンカーのためにパスを解放します）。CI ではコンパイルが通っており、実機での検証は進行中です。
- 外向きの JSON はすべて、不正な UTF-8 を例外ではなく U+FFFD に置き換えてダンプされます——ローカライズされたツールチェーンが CP932 のビルド出力をログへ流し込んでも、スーパーバイザーをクラッシュさせてはならないからです。

## 設計上の決定

**`trusscli` と分離する。** 開発 CLI の仕事はプロジェクトのビルドであり、見守って手元へ報告するような振る舞いからは切り離しておきます——npm と pm2 の分業と同じです。おせっかいはビルダーにとって鬱陶しく、運用エージェントにとってはそれこそが価値のすべてで、両者の対象はほとんど重なりません（ワークショップの初心者が AnchorBolt に出会うことはなく、インスタレーションのプロは2つ目のツールを厭いません）。

**データベースではなく、ファイル。** 履歴は必須要件なので、ストアは常に存在します。追記専用の JSONL ＋タイムスタンプ付きの JPEG は、クラッシュに強く、マイグレーション不要で、grep でき、デーモンも要りません。データベース（FTS インデックスとしての SQLite）は、長期間にわたる検索に対する*後々の*最適化として考えうるものであって、ファイルを信頼できる情報源に据えたままにします——初日からの依存物ではありません。

**OAuth ではなく、トークン。** オフライン動作は必須要件です——不安定なネットワーク上の会場も動き続けなければならず、サーバーは Google/SSO への依存なしにプロジェクトごとにデプロイできなければなりません。ハッシュ化されたトークンは、即時の失効（ここではステートレス性より重要です）と、オフラインでのハッシュ照合をもたらします。SSO がどうしても必要になった場合は、AnchorBolt の中ではなくリバースプロキシで終端します。

**サービスごとのアダプターではなく、1つのテンプレート化された Webhook エンジン。** Slack、Discord、ntfy、Kuma、そして任意の汎用エンドポイントは、いずれもボディテンプレートを持つ同じ HTTP POST であり、プリセットはそれを事前に埋めるだけです。サービスを追加するとは、アダプターを追加することではなく、プリセットを追加することです。同じエンジンが会場マシン上（`anchorbolt.json` のアプリごとの `notify`）とサーバー上（`config/notify.json` のフリート全体の通知——ダッシュボードの Notify タブ）の両方で動きます。

**通知は一方向で、対話はダッシュボードに置く。** Webhook は外へ押し出しますが（集約・重複除去済み）、変更を伴うアクションの承認、アプリの操作、ログの閲覧——それらはダッシュボードで起こることで、チャットアプリの中では決して起こりません。その道は、サービスごとのボットと OAuth の泥沼へと続いています。

**ローカルファースト。** ほとんどのアプリが必要とするのは自動再起動とローカルログだけで、サーバーはフリートを運用する少数のためのものです。すべてのサーバー機能はオプションです——素の `anchorbolt start` は、それ自体で完結した、役立つツールです。
