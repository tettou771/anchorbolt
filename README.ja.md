# AnchorBolt

**どこか別の場所で動かしっぱなしのアプリを、しっかり見守る。**

![フリートのウォール — すべてのアプリをひと目で](docs/images/wall.png)

## なぜ必要か

アートインスタレーション、美術館の展示、イベントブースのためにアプリを作った——そして今それは、別の建物、別の街のマシンで、何週間も無人のまま動き続けている。そんなある日、深夜2時にクラッシュする。あるいは静かにハングする。あるいはクライアントから変更を頼まれるが、現地まで足を運ぶのは気が進まない。よくある対処法といえば、シェルスクリプトの寄せ集めと cron ジョブ、そして「念のため」に誰かがスマホのカメラを画面に向けること、くらいのものです。

**AnchorBolt は、そうしたインスタレーションのための運用レイヤーです。** 会場のマシン上であなたのアプリを見守り——死んだりハングしたら再起動し、ログを収集し、ライブサムネイルを手元へ送り返します。だから、自分のデスクからすべてのアプリをひと目で確認できます。人の手が必要になったときには、再起動やアップデート、さらにはブラウザからアプリを遠隔操作することもできます。この中核部分を使うのに、アプリ側のコード変更は**一切不要**です。

[TrussC](https://github.com/TrussC-org/TrussC) アプリ向けに作られており（ヘルスチェックとスクリーンショットに TrussC の MCP エンドポイントを使います）、監視・ログ収集・通知の部分は、あなたが起動できるどんなプログラムに対しても機能します。

## できること

- **自動再起動** — アプリを2通りの方法で監視し（終了した／無反応になった）、復帰させます。アプリ側のコードは不要です。
- **ウォール** — すべてのアプリのライブサムネイルを1つのブラウザページに集約。緑／赤でひと目で状態がわかり、タブごとにグループ化されます。
- **ログを、常に** — アプリのログをローカルで収集し、サーバーへ送ります。数時間のネットワーク断も乗り越え、中断したところから再接続します。
- **通知** — クラッシュ、ハング、あるいはアプリが上げたアラートが、Slack / Discord / ntfy / Uptime Kuma に届いたり、ウォール上に赤いバッジとして表示されたりします。
- **リモートアップデート** — ボタンを押すだけで、アプリを動かしたまま会場マシン上で `git pull` ＋リビルドを実行します。ビルドが成功したときだけ新しいバイナリに切り替わり、新しいバイナリが起動しなければ自動でロールバックします。
- **ライブビュー ＋ リモートコントロール** — アプリの画面をライブで見られ、許可すればブラウザからクリックやキー入力もできます。
- **グラフとステータス** — fps、メモリ（アプリ自身のものとマシン全体のもの）、そしてアプリが公開することを選んだ任意の数値／画像を、時系列でプロットします。
- **AI 向け** — サーバー自体が MCP エンドポイントなので、アシスタントが昨夜のログを検索し、スクリーンショットを取得し、（人間の承認を経て）アプリを再起動できます。
- **アクセス制御** — ロール付きのオペレーターログイン、クライアントごとの表示範囲スコープ、そして長い秘密文字列を持ち回らずに済む6桁コード。

## 1つのツール、2つの役割

AnchorBolt は単一のバイナリです。**会場のマシン**ではスーパーバイザーを、**あなたのサーバー**ではフリートダッシュボードを動かします。プロトコルの両端が同じコードベースに含まれているため、両者のバージョンがずれることは決してありません。

```bash
# on the venue machine — supervise the app, phone home
anchorbolt start -p myApp --server https://ops.example.com

# on your server — the dashboard everyone looks at
anchorbolt serve
```

ローカルの自動再起動だけで十分なアプリは、`--server` をまるごと省略できます——スーパーバイザーは単体でも問題なく動きます。

## はじめる

[Get Started ガイド](docs/GET_STARTED.ja.md) は、30秒のローカルお試しから、Cloudflare トンネルの背後での実運用デプロイまでを案内します——アプリのグループ化、Slack の連携、クライアントへの読み取り専用ログインの付与、リモートコントロールの有効化。どのステップもコピー＆ペーストで進められます。

```bash
# the 30-second taste (two terminals on one machine)
anchorbolt serve                       # open http://localhost:54722/
anchorbolt start -p myApp --server http://localhost:54722
```

## どう作られているか

[Architecture ドキュメント](docs/ARCHITECTURE.ja.md) では設計を解説します：2層のスーパービジョンモデル、DB を使わないファイルストレージ、オフラインに強いログ配送カーソル、2種類のトークン、WebSocket コマンドチャンネル、フリート MCP、そしてそれぞれの選択の理由（なぜ `trusscli` と分離しているのか、なぜデータベースではなくファイルなのか、なぜ OAuth ではなくトークンなのか）。

## ステータス

**macOS と Linux** での実運用に必要な機能は揃っています。会場側のスーパーバイザーは **Windows** 向けにもビルドできます（実機での検証は進行中）。まだ若く、速いペースで進化しているツールです——CLI とダッシュボードはデプロイできる程度に安定しており、最近の追加（AI 主導の変更に対する承認キュー、フリート全体の通知、共有可能な閲覧専用 URL）もすべて追加的なレイヤーとして実装されました。

意図的に **`trusscli` とは別のツール**にしています：開発 CLI はプロジェクトをビルドし、AnchorBolt は本番でそれらを見守ります。確定した設計の全体像は TrussC の ROADMAP（kiosk / fleet の項目）を参照してください。

## ビルド

標準的な TrussC プロジェクトです。`tc_get_health` / `tc_get_screenshot` / `TRUSSC_LOG_FILE` を備えた TrussC core（v0.7.0 以降）が必要です。

```bash
trusscli update   # regenerate CMakeLists/CMakePresets (they are gitignored)
trusscli build
```
